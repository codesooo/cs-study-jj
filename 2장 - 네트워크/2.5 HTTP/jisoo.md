## intro
**HTTP**는 **애플리케이션 계층**으로, **웹 서비스 통신**에 사용된다.

## 2.5.1 HTTP/1.0

### RTT 증가
<img src="https://github.com/codesooo/cs-study-jj/assets/88030238/33051783-f70a-4239-9c37-f26d35de3fee" width="60%">

서버로부터 파일을 가져올 때마다 TCP의 3-way handshake를 계속해서 열어야 함 <br>
-> RTT 증가

### RTT의 증가를 해결하기 위한 방법
#### 문제점
- RTT 증가 -> 서버에 많은 부담, user 응답 시간 증가

#### 해결방안
#### 이미지 스플리팅
많은 이미지가 합쳐진 하나의 이미지를 다운받고, 이를 기반으로 background-image의 position을 이용하여 이미지를 표기
#### 코드 압축
코드를 압축하여 개행 문자와 빈칸을 없애서 코드의 크기를 최소화
#### 이미지 Base64 인코딩
이미지 파일을 64진법으로 이루어진 문자열로 인코딩 <br>
- 장점 : 서버와의 연결을 열어서 이미지에 대한 HTTP 요청할 필요가 없음
- 단점 : Base64 문자열로 변환할 경우 크기가 37% 정도 더 커짐
  

## 2.5.2 HTTP/1.1
- 한 번 TCP 초기화를 한 이후에 여러 개의 파일을 송수신 (by. keep-alive)
  - 매번 TCP를 연결하는 HTTP/1.0과의 차이
 
<img src="https://github.com/codesooo/cs-study-jj/assets/88030238/b9705d52-f461-430a-a986-13367cdfa956" width="60%">

한 번 TCP 3-way handshake가 발생하면 그 다음부터 발생하지 않는다.<br>
But, 문서 안에 포함된 다수의 리소스를 처리하려면 요청할 리소스 개수에 비례해서 대기 시간이 길어진다.

### HOL Blocking (Head of Line Blocking)
- 네트워크에서, 같은 큐에 있는 패킷이 그 첫 번째 패킷에 의해 지연될 때 발생하는 성능 저하 현상

### 무거운 헤더 구조
HTTP/1.1의 헤더에는 쿠키 등 많은 메타데이터가 들어있고, 압축되지 않아 무겁다.

## 2.5.3 HTTP/2
- SPDY 프로토콜에서 파생된 HTTP/1.x 와의 차이
  - 지연 시간을 감소
  - 응답시간 더 빠르게
  - 멀티플렉싱, 헤더 압축, 서버 푸시, 요청의 우선순위 처리를 지원
  
### 멀티플렉싱
- 여러 개의 스트림을 사용하여 송수신하는 것
- 특정 스트림의 패킷이 손실되어도 해당 스트림에만 영향을 미침. 나머지 스트림은 멀쩡하게 동작 가능!
- 애플리케이션에서 받아온 메시지를 독립된 프레임으로 조각내어 서로 송수신한 이후 다시 조립하여 데이터를 주고받음
  - 단일 연결을 사용하여 병렬로 여러 요청 및 응답 가능
  - -> **HOL Blocking** 해결 가능

### 헤더 압축
- **`HTTP/2`** 는 헤더 압축을 통해 HTTP/1.x의 크기가 큰 헤더 문제 해결
- HPACK 압축 형식을 가짐
  - HPACK : 허프만 코딩 압축 알고리즘 사용

#### 허프만 코딩
- 문자열을 문자 단위로 쪼갬
- 빈도수 측정
  - 빈도가 높은 정보 : 적은 비트 수로 표현
  - 빈도가 낮은 정보 : 많은 비트 수로 표현
- 결과 : 데이터 표현에 필요한 비트양을 줄임

### 서버 푸시
**`HTTP/2`** 는 클라이언트 요청 없이 서버가 바로 리소스 푸시 가능

## 2.5.4 HTTPS
**`HTTPS`** : 애플리케이션 계층과 전송 계층 사이에 SSL/TLS 계층(신뢰 계층)을 넣은 신뢰할 수 있는 HTTP 요청<br>
-> 통신 암호화

<img src="https://github.com/codesooo/cs-study-jj/assets/88030238/2db0c6be-7281-4589-9cbe-0fa888b2fdbf" width="40%">

### SSL/TLS
- 전송 계층에서 보안을 제공하는 프로토콜
- 클라이언트 - 서버 통신에서 제3자의 메시지 도청 및 변조 방지
- 네크워크 상의 인터셉터 방지
  - 인터셉터 : 공격자가 서버인 척하며 user 정보를 가로채킄 것
- 보안 세션을 기반으로 데이터 암호화
  - 보안 세션이 만들어질 때 인증 메커니즘, 키 교환 암호화 알고리즘, 해싱 알고리즘이 사용됨

#### 보안 세션
- 보안이 시작되고 끝나는 동안 유지되는 세션
- SSL/TLS는 handshake를 통해 보안 세션을 생성하고 이를 기반으로 상태 정보 등을 공유함
- 과정
  - 클라이언트 : 서버로 사이퍼 슈트를 전달
  - 서버 : 받은 사이퍼 슈트의 암호화 알고리즘 리스트가 제공 가능한지 확인
    - 제공 가능하다면 : 인증 메커니즘이 시작되고, 이후에 해싱 알고리즘 등으로 암호화된 데이터의 송수신이 시작됨
      - 인증 메커니즘 : 서버 -> 클라이언트로 인증서를 보내는 것
  
##### 사이퍼 슈트
- **`프로토콜`** , **`AEAD 사이퍼 모드`** , **`해싱 알고리즘`** 이 나열된 규약

##### AEAD 사이퍼 모드
- AEAD : Authenticated Encryption with Associated Data
#### 인증 메커니즘
- CA(Certificate Authorities)에서 발급한 인증서를 기반으로 이루어짐
- CA에서 발급한 인증서
  - 공개키를 클라이언트에 제공
    - 공개키 : 안전한 연결을 시작하기 위해 필요
  - 접속한 서버가 신뢰할 수 있는 서버임을 보장
  - 인증서는 서비스 정보, 공개키, 지문, 디지털 서명 등으로 이루어져 있음
- CA는 신뢰성이 엄격하게 공인된 기업들만 참여 가능
  - ex) Comodo, GoDaddy, Globalsign, 아마존

##### CA 발급 과정
- 서비스에 대한 CA 인증서를 발급받으려면 다음을 CA에 제출해야 함
  - 사이트 정보
  - 공개키
- 이후 CA가 지문(=공개키를 해시한 값)을 사용하는 CA의 비밀키 등을 기반으로 CA 인증서 발급

#### 암호화 알고리즘
- 키 교환 암호화 알고리즘 (디피-헬만 방식을 근간으로 만들어짐)
  - 대수곡선 기반의 ECDHE(Eliptic Curve Diffie-Hellman Ephermeral)
  - 모듈식 기반의 DHE(Diffie-Hellman Ephermeral)
  
##### 디피-헬만 키 교환 알고리즘
- 암호키를 교환하는 방법 중 하나
 $$y=g^xmodp$$
- g,x,p를 알면 y를 구하기 쉬움
- g,y,p를 알면 x를 구하기 어려움

#### 해싱 알고리즘
- 데이터를 더 작고, 섞여 있는 조각으로 만드는 알고리즘

##### SHA-256 알고리즘

- 비트 코인을 비롯한 많은 블록체인 시스템에서 사용함
- 전처리를 하고 전처리된 메시지를 기반으로 해시를 반환
  - ex) 해싱을 해야 할 메시지에 1을 추가하는 등

## 2.5.5 HTTP/3
- WWW에서 정보를 교환하는데 사용되는 HTTP의 세번째 버전
- QUIC 계층 위에서 돌아감
- UDP 기반
- 멀티플렉싱 보유
- 초기 연결 설정 시 지연 시간 감소

### 초기 연결 설정 시 지연 시간 감소
- QUIC는 TCP를 사용하지 않으므로 통신을 시작할 때 3-way handshake 과정을 거치지 않아도 됨
- QUIC는 첫 연결 설정에 1-RTT만 소요됨
  - 클라이언트가 서버에 어떤 신호를 한번 주고, 서버도 거기에 응답하기만 하면 바로 본 통신 시작 가능
- QUIC는 순방향 오류 수정 메커니즘(FEC, Forword Error Correction)이 적용됨
  - FEC
    - 전송한 패킷이 손실되었다면, 수신측에서 에러를 검출하고 수정하는 방식
    - 열악한 네트워크 환경에서도 낮은 패킷 손실률을 자랑함

## 용어 정리
#### RTT
- 패킷이 목적지에 도달한 뒤에 다시 출발지로 돌아오기까지 걸리는 시간
- 패킷 왕복 시간

#### 인코딩
정보의 형태나 형식을 표준화, 보안, 처리 속도 향상, 저장 공간 절약 등을 위해 다른 형태나 형식으로 변환하는 처리 방식

#### 스트림 stream
시간이 지남에 따라 사용할 수 있게 되는 일련의 데이터 요소를 가리키는 데이터 흐름

#### 세션
- 운영체제가 어떠한 user로부터 자신의 자산 이용을 허락하는 일정한 기간
- user는 일정 시간 동안 응용 프로그램, 자원 등을 사용할 수 있음

#### 개인키 (=비밀키)
- 개인이 소유하고 있는 키
- 반드시 자신만이 소유해야 하는 키

#### 공개키
공개되어 있는 키

#### 해시
다양한 길이를 가진 데이터를 고정된 길이를 가진 데이터로 매핑한 값

#### 해싱
- 임의의 데이터를 해시로 바꿔주는 일
- 해시 함수가 이것을 담당

#### 해시 함수
임의의 데이터를 입력으로 받아 일정한 길이의 데이터로 바꿔주는 함수
