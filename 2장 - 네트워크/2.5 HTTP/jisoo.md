## intro
**HTTP**는 **애플리케이션 계층**으로, **웹 서비스 통신**에 사용된다.

## 2.5.1 HTTP/1.0

### RTT 증가
<img src="https://github.com/codesooo/cs-study-jj/assets/88030238/33051783-f70a-4239-9c37-f26d35de3fee" width="60%">

서버로부터 파일을 가져올 때마다 TCP의 3-way handshake를 계속해서 열어야 함 <br>
-> RTT 증가

### RTT의 증가를 해결하기 위한 방법
#### 문제점
- RTT 증가 -> 서버에 많은 부담, user 응답 시간 증가

#### 해결방안
#### 이미지 스플리팅
많은 이미지가 합쳐진 하나의 이미지를 다운받고, 이를 기반으로 background-image의 position을 이용하여 이미지를 표기
#### 코드 압축
코드를 압축하여 개행 문자와 빈칸을 없애서 코드의 크기를 최소화
#### 이미지 Base64 인코딩
이미지 파일을 64진법으로 이루어진 문자열로 인코딩 <br>
- 장점 : 서버와의 연결을 열어서 이미지에 대한 HTTP 요청할 필요가 없음
- 단점 : Base64 문자열로 변환할 경우 크기가 37% 정도 더 커짐
  

## 2.5.2 HTTP/1.1
- 한 번 TCP 초기화를 한 이후에 여러 개의 파일을 송수신 (by. keep-alive)
  - 매번 TCP를 연결하는 HTTP/1.0과의 차이
 
<img src="https://github.com/codesooo/cs-study-jj/assets/88030238/b9705d52-f461-430a-a986-13367cdfa956" width="60%">

한 번 TCP 3-way handshake가 발생하면 그 다음부터 발생하지 않는다.<br>
But, 문서 안에 포함된 다수의 리소스를 처리하려면 요청할 리소스 개수에 비례해서 대기 시간이 길어진다.

### HOL Blocking (Head of Line Blocking)
- 네트워크에서, 같은 큐에 있는 패킷이 그 첫 번째 패킷에 의해 지연될 때 발생하는 성능 저하 현상

### 무거운 헤더 구조
HTTP/1.1의 헤더에는 쿠키 등 많은 메타데이터가 들어있고, 압축되지 않아 무겁다.

## 용어 정리
#### RTT
- 패킷이 목적지에 도달한 뒤에 다시 출발지로 돌아오기까지 걸리는 시간
- 패킷 왕복 시간

#### 인코딩
정보의 형태나 형식을 표준화, 보안, 처리 속도 향상, 저장 공간 절약 등을 위해 다른 형태나 형식으로 변환하는 처리 방식
